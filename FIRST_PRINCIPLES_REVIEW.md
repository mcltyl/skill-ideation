# Skill Ideation - 第一性原理評估

## 步驟一：識別目的

**這個 skill 最終要達成什麼？**

表面目的：生成 skill ideas
真正目的：**幫助人找到值得建立的 skill，讓 AI agent 更有用**

更深層：
- 擴展 agent 的能力邊界
- 讓知識可複用、可傳遞
- 填補「人需要但沒有現成解答」的 gap

## 步驟二：拆解到基本事實

**關於 skill ideation，確定為真的是什麼？**

1. **一個好 skill = 解決真實問題** — 不是「聽起來酷」
2. **Narrow > Broad** — 範圍小但做好，比範圍大但空泛有價值
3. **Teachable** — 必須能用文字傳達，不是只有「你得自己體會」
4. **Timeless > Timely** — 框架比資料持久
5. **Findable** — 使用者想到要找的時候能想到這個 skill 存在
6. **Context-rich** — 不只是 what，還有 why 和 when

## 步驟三：識別隱藏假設

**我正在接受哪些未經檢驗的假設？**

| 假設 | 檢驗 |
|------|------|
| 「主題越多越好」 | ❌ 不對。數量不是價值，重要的是命中率 |
| 「大主題更有價值」 | ❌ 不對。Narrow 往往更有用 |
| 「分類要整齊」 | ❓ 不一定。分類是給人看的，但 ideas 不需要整齊才有用 |
| 「我應該覆蓋所有領域」 | ❌ 不對。不需要全覆蓋，需要深度和相關性 |
| 「Ideas 要很完整才能提出」 | ❌ 不對。半成形的 idea 也有價值，可以被發展 |

## 步驟四：重新推導

**如果從基本事實出發，skill ideation 應該怎麼做？**

### 核心原則

1. **從問題出發，不是從分類出發**
   - 不是「Travel 類可以有什麼」
   - 而是「旅行時人會遇到什麼問題」

2. **從情境出發，不是從概念出發**
   - 不是「時間管理」
   - 而是「為什麼我又在 deadline 前一天才開始」

3. **從動詞出發，不是從名詞出發**
   - 不是「溝通」
   - 而是「說不」「道歉」「請求幫助」

4. **從失敗出發**
   - 「人在什麼情況下會搞砸」
   - 「什麼事情人們一再犯錯」

5. **從沒人教出發**
   - 「學校不教但每個人都需要」
   - 「父母沒說但早知道就好」

### 生成方法重構

| 舊方法 | 新方法 |
|--------|--------|
| 從分類想 idea | 從情境/問題想 idea |
| 列出名詞 | 列出動詞 + 情境 |
| 「什麼主題還沒覆蓋」 | 「人在什麼時刻需要幫助」 |
| General 概念 | Narrow 瞬間 |

### 品質指標重構

一個好 idea 應該能回答：

1. **誰會需要這個？** — 有明確的人
2. **在什麼情境？** — 有明確的 when
3. **需要的瞬間是什麼？** — 有明確的 trigger
4. **他們現在怎麼解決？** — 有替代方案（可能是爛的）
5. **這個 skill 能給什麼？** — 有明確的 outcome

## 步驟五：驗證推導

**推導中最脆弱的環節是什麼？**

1. **假設人會找 skill** — 不一定。很多 skill 存在但沒人用
   → 解法：需要 「findability」設計，名字要直覺

2. **假設 narrow 就是好** — 不完全。太 narrow 可能沒有足夠價值
   → 解法：Narrow 但 universal（很多人會遇到的 narrow 情境）

3. **假設 ideas 會被執行** — 不一定。產生 idea 容易，執行難
   → 解法：Ideas 要包含「為什麼值得做」的說服

---

## 修正行動

### 1. 重組 skill 結構

原本：按大分類列 ideas
修正：按「情境類型」+ 「觸發時刻」組織

### 2. 增加品質篩選器

每個 idea 要能通過：
- [ ] 有明確的人 (who)
- [ ] 有明確的時刻 (when)  
- [ ] 不是太 obvious（Google 不到好答案）
- [ ] 不是太 niche（只有 10 個人需要）

### 3. 增加「why worth building」

不只是 idea，還要有「為什麼這個值得做」

### 4. 重新命名

把 Generic 名字改成情境驅動：
- ❌ `time-management`
- ✅ `deadline-panic-recovery`

---

## 結論

**Original skill 的問題：**
- 太多 category-driven ideas
- 缺少情境具體性
- 沒有品質篩選

**修正後的核心改變：**
- 從「分類覆蓋」→「問題驅動」
- 從「名詞」→「動詞 + 情境」
- 從「數量」→「品質 + why」
